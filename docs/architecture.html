01Visão geral

O repositório é um hub de projetos de dados. A página inicial (index.html) lê um manifesto (projects.json) e monta os cards dos projetos, agrupados por abordagem (Visualização, Qualidade, Pipeline/ETL, Predição). O projeto principal é Conexões da História: um grafo de pessoas, lugares e fatos de Recife/Pernambuco, com visualizações (diagrama, mapa, matriz), lista e um painel de qualidade (fill rate).

O dado é somente leitura em runtime: nada é gravado pelo site. A edição acontece numa planilha; o site apenas consome arquivos gerados.

02Padrão de arquitetura

Combinação de padrões conhecidos:

  • Static-first / Jamstack (sem o “A” de API): tudo é pré-construído e servido como arquivo estático; renderização client-side (D3, Leaflet, canvas).
  • ETL em build-time: extract da planilha → transform/validate em Python → load como JSON.
  • Single source of truth → derivados (precomputed views): os JSON são views geradas e commitadas; um drift-check garante fonte == derivado.
  • Flat-file data (schema-on-write): sem banco; integridade validada na escrita/build.
  • MPA + roteamento por viewport: múltiplas páginas, com troca desktop↔mobile em 1024 px.
  • Registry/manifest: projects.json dirige o hub.

Integridade

  • Erros (falham o build/CI): id duplicado, nome duplicado, origem de aresta não resolvida.
  • Avisos: aresta quebrada (alvo não resolvido), alias sem canônico, self-loop, descrição/cidade faltando.
  • Drift-check diário: se o CSV commitado divergir da planilha, o CI falha.

05Frontend

Desktop — pages/

Páginas ricas em D3.js (diagrama) e Leaflet + tiles CARTO (mapa geográfico). i18n via páginas pages/en/.

Mobile — mobile/site/

App self-contained (app.js/app.css, fonte self-hosted). Algumas telas (mapa, diagrama) são páginas próprias com canvas e i18n embutido.

Roteamento por viewport

Cada página tem um par desktop↔mobile. Abaixo de 1024 px o desktop redireciona para o mobile; em ≥1024 px o mobile redireciona para o desktop — nos dois sentidos, e reagindo a resize.

i18n (PT/EN)

Strings estáticas marcadas com data-i18n; um dicionário aplica o EN quando localStorage.lang === "en". Descrições de conteúdo têm sua própria fonte de tradução.

08Camada compartilhada

Para reduzir duplicação entre desktop e mobile (ADR 0002), shared/ é a fonte única do que ambos usam. Já existe tokens.json (cores por tipo/categoria) → build_shared.py gera tokens.css (com --check no CI). Próximos passos: dicionário i18n e modelo de dados do grafo compartilhados.

09Decisões de arquitetura (ADRs)

Registros datados em docs/adr/ (Contexto · Situação anterior · Decisão · Benefícios · Pontos de atenção · Melhora esperada).

01
Sem banco de dados — JSON estático gerado a partir da planilha. Um banco só se justificaria com uma camada de API/serviços (projeto à parte). ler →
02
Unificar a camada, não a página — compartilhar dados/i18n/tokens entre desktop e mobile, mantendo a apresentação separada por formato. ler →
03
i18n no modelo de dados (sem SQL) — PT e EN como cidadãos de primeira classe; relatório de cobertura no lugar de uma view SQL. ler →

10Stack

  • Dados/build: Python (biblioteca padrão), CSV, JSON.
  • Frontend: HTML/CSS/JS vanilla; D3.js v4; Leaflet + tiles CARTO; canvas/SVG.
  • Fonte da verdade: Google Sheets (publicado como CSV).
  • Hospedagem/CI: GitHub Pages + GitHub Actions.

11Quando revisitar

A decisão de “sem banco” vale para o site estático. Reconsiderar se surgir:

  • uma camada de API/serviços (runtime dinâmico, escrita, auth) — seria um projeto separado, com o próprio banco e ADR;
  • crescimento do dataset além do que o JSON estático serve bem, ou necessidade de consulta real no cliente — avaliar SQLite em build-time ou SQLite no browser (sql.js) antes de qualquer banco em servidor.
