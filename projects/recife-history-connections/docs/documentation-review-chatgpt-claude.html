Recommended Improvements

1. Split the root README

Where: README.md

  • Keep: project intro, run locally, data source, rebuild commands, links.
  • Move: technology upgrade paths to docs/technology-options.md.
  • Why: the README is currently the largest markdown file and mixes onboarding with speculative future options.

2. Refresh mobile status

Where: mobile/README.md

  • Resolve contradictions like "Map projection is out of scope" versus existing mobile map work.
  • Replace phase chronology with "Current architecture", "Build", "Quality targets", and "Known debt".
  • Why: stale status text makes PR reviews harder because agents may optimize against old constraints.

3. Keep implementation prompt as archive

Where: docs/implementation-prompt.md

  • Add a short banner: "Historical implementation prompt; current runbooks live in README/mobile/docs."
  • Move active constraints into smaller operational docs if they are still enforced.
  • Why: at 306 lines, it is too long for active navigation but valuable as design context.

4. Make PR review docs canonical

Where: docs/pr-review-flow.md and docs/pr-review-flow.html

  • Keep the markdown as the source of process truth.
  • Use the HTML as the operator-friendly presentation.
  • Why: this prevents drift between the runbook and the visual page.

5. Normalize generated documentation headers

Where: dashboard/dashboard.html, Quality/modules-features-matrix/*.html

  • Add visible "Generated from ..." notes when missing.
  • Include regeneration command and source JSON links near the top.
  • Why: generated files can be large; readers need to know not to hand-edit them.

6. Split wide operational tables

Where: docs/task-evolution.md, Quality/bug-fixes.md

  • Consider JSON/CSV as source of truth plus generated HTML/table views.
  • Keep markdown as a summary, not the whole database.
  • Why: the line count is small, but line width is extreme: up to 709 and 893 characters.
