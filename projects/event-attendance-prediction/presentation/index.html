03

Motivation

Raw RSVP overstates who actually shows up

  • Facebook events collect Going / Interested — but real attendance is a fraction, and the gap widens with size.
  • Organizers plan staff, security, food, restrooms from unreliable signals.
  • Over- or under-provisioning is costly — a decision-support gap.
04

What it's based on

A 2016 master's system + real digitized data

  • The author's article "EVE — Estimation of Variables to Events" (C.E.S.A.R, Recife).
  • 20 events digitized from the article: 16 public events by size (tables 7–10) + a 4-rehearsal field experiment (May 2013, a Recife percussion group).
  • The article's own arithmetic inconsistencies are preserved and asserted in tests, not silently "fixed".
05

What the system proposes

Learn attendance rates — don't guess them

  • EVE's old weakness: an operator guessed the attendance proportion per response type.
  • Now: learn conversion rates per response type × size class from data.
  • Output: an estimate, a likely range, and a decomposition — adjustable by external effects (rival match, rain… as ± %).

attendance ≈ Σ ( counttype × rate[size][type] ) × effects

The opportunity

Better planning — and a model that improves with use

  • Better resource planning & ROI; fewer over/under-provisioning mistakes.
  • A recalibration loop: log real attendance → export → refit → improved rates embedded back.

08

Test architecture

Automated tests + a matrices spine

  • pytest (22): formula units · data-integrity (schema, sums, and the article's known inconsistencies) · model (rate order, bounds) · regression pinning exact metrics.
  • Component matrices as the test/risk spine: an operation flow (each step = a scenario) + module-feature, cross-dependence, reuse, and identification (selector) matrices.
  • Manual UI cases for tool & dashboard, mapped to data-testid roots.
  • Docs stay in sync: a check script + pre-commit hook + a /sync-matrices skill.
