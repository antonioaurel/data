Event attendance prediction · EVE

Estimating real attendance for Facebook events

A modern reimplementation of a 2016 master's system as a data and modeling project. It turns unreliable RSVP counts into a calibrated turnout estimate.

03

Motivation

Raw RSVP overstates who actually shows up

  • Facebook events collect Going and Interested, but real attendance is only a fraction, and the gap widens with size.
  • Organizers plan staff, security, food and restrooms from unreliable signals.
  • Over provisioning wastes money; under provisioning breaks the event.
Although EVE dates to 2016, this is a today's problem: platforms still expose Going / Interested, and organizers still plan against ghosts.

04

What it's based on

A 2016 master's system plus real digitized data

  • The author's article "EVE, Estimation of Variables to Events" (C.E.S.A.R, Recife).
  • 20 events digitized from the article: 16 public events by size (tables 7 to 10), plus a 4 rehearsal field experiment (May 2013, a Recife percussion group).
  • Real turnout is known for every event, so predictions can be backtested.
05

What the system proposes

Learn attendance rates, don't guess them

  • Learn conversion rates per response type and size class from data.
  • Output an estimate, a likely range, and a decomposition of where the number comes from.
  • Adjust for external effects (a rival match, rain) as a plus or minus percentage.

The opportunity

Better planning, and a model that improves with use

  • Better resource planning and ROI; fewer over or under provisioning mistakes.
  • A recalibration loop: log real attendance, refit, embed the improved rates back.

Test architecture

Automated tests plus a matrices spine

  • pytest (22): formula units, data integrity (schema, sums, and the article's known quirks), model checks, and regression pinning exact metrics.
  • Component matrices as the risk spine: an operation flow (each step is a scenario) plus feature, dependency, reuse and selector matrices.
  • Manual UI cases mapped to data-testid roots.
  • Docs stay in sync via a check script, a git hook, and a skill.

Key decisions · ADRs (1 of 2)

Scope and platform

Analysis, not a rebuilt app
Why EVE was a requirements era web system; rebuilding it adds no research value, and its live data source no longer exists.
Tradeoff no production tool for organizers yet; the deliverable is the model plus a working demo.
No Facebook Graph API
Why the endpoints that imported guest lists and RSVPs were removed in 2018, after Cambridge Analytica; live import is impossible.
Tradeoff input is manual or aggregated counts, not an automatic feed.
No backend or database
Why with about 20 events, CSVs in git already give storage, history, diff and review for free.
Tradeoff no multi user persistence; the tool keeps history in the browser plus a CSV export.
Python for the offline tier
Why a natural fit for tabular modeling and backtesting; notebook native, testable, light on dependencies.
Tradeoff two languages (Python offline, JS client), bridged by the coefficient file.

Key decisions · ADRs (2 of 2)

Model and interface

Learn rates from data
Why EVE's core weakness was operator guessed proportions; learning them removes subjectivity and enables the recalibration loop.
Tradeoff needs data; v1 is an in sample calibration on 20 events.
Static HTML and vanilla JS
Why the pages are small and must run with no backend; a framework and build step add weight for no gain.
Tradeoff more manual DOM code, mitigated by the component matrices.
Coefficient file as the contract
Why one small versioned artifact decouples the model from both pages; a refit repaints them at once.
Tradeoff a key change breaks both consumers, so it is tested as a contract.
Dashboard is three pane
Why chosen from options; one command center view of the event list, charts and metrics, in a single light theme.
Tradeoff a denser layout that needs screen width.
Tool is a single page form
Why all steps visible on one vertical page; Next scrolls to the next and is gated on mandatory fields.
Tradeoff a longer page, mitigated by the sticky stepper.

Current state

Built and verified
